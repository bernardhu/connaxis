# 路线图

这份路线图的目标，是让贡献者更容易看清项目当前真正优先推进的事情。
它将近期主线工作与实验性探索拆开，避免把“会持续推进的事项”和“值得研究但暂未承诺的方向”混在一起。

## 主线优先事项

这些是项目近期预期会持续推进的方向。

### 1. 公开 API 边界与更小的接口集

将面向外部的支持边界稳定在 `connaxis`、`connection`、`eventloop`、`evhandler` 和 `websocket` 这几个包上，同时尽量把接口集收紧到“刚好覆盖需求”的最小范围。

完成后的状态应当是：

- README、示例和包级文档指向同一套受支持入口
- 公开行为可以通过文档理解，而不要求用户先理解内部 wiring
- 更小的接口集仍然能覆盖所需的 handler 与协议集成路径，而不是无谓扩大 API
- 示例、配置说明与当前实际行为保持一致

适合的贡献类型：

- 文档和示例修正
- 包级文档补充
- 能在不削弱已支持行为的前提下收紧接口面的窄范围 API 简化

### 2. 面向 BoringSSL / TLS Fuzzer 的验证与兼容性

通过 `bogo`、`tlsfuzzer` 等验证套件，以及必要的 focused regression tests，提升协议处理行为的可信度。

完成后的状态应当是：

- `bogo`、`tlsfuzzer` 以及相关 TLS 验证流程保持可运行且有实际价值
- 对于套件未覆盖的握手和关闭路径边界情况，有 focused tests 兜住
- 兼容性缺口要么有测试覆盖，要么有明确说明

适合的贡献类型：

- 围绕 `bogo`、`tlsfuzzer` 和相关 TLS 流程的验证脚本改进
- 兼容性说明和可复现性修正
- 补足外部套件未覆盖部分的 focused tests

### 3. 跨 Linux 内核版本的 TLS / kTLS 验证

项目的主要支持平台是 Linux。对于 macOS 和 BSD，目标是保证基础链路可跑通，而不是追求与 Linux 同等级的特性覆盖。

完成后的状态应当是：

- TLS / kTLS 行为在更多 Linux 内核版本上得到验证
- 内核版本相关的兼容性缺口要么有测试覆盖，要么有明确说明
- 非 Linux 路径可用于基础验证，但不扩大主支持范围

适合的贡献类型：

- Linux 内核矩阵验证
- 内核约束、限制和预期行为的文档补充
- 非 Linux 平台“支持到什么程度”的可复现说明

### 4. 验证与基准可复现性

让项目的质量和性能结论更容易在干净环境中复现。

完成后的状态应当是：

- benchmark 和验证脚本有现实可用的前置条件说明
- Linux 测试环境和 TLS 工作流保持可复现
- 性能报告来自可理解、可重复的输入过程

适合的贡献类型：

- benchmark 脚本整理
- 可复现性修正
- 报告和方法学文档改进

## 实验性方向

这些方向值得探索，但不是当前主线承诺。
涉及这些方向的变更应先以实验、原型或隔离分支方式推进，再决定是否进入生产 runtime。

### `io_uring` 后端探索

目标：

- 评估 Linux `io_uring` 路径是否能在当前 poll-based runtime 之外带来清晰收益

进入主线前至少应满足：

- 在目标负载上有可测量收益
- 复杂度仍然可理解、可审阅
- 故障处理和所有权语义保持显式

### arena 风格内存分配实验

目标：

- 测试窄范围、短生命周期的分配路径是否适合 arena 风格管理

进入主线前至少应满足：

- 在测量到的热点路径上降低分配开销
- 对象所有权和回收时机依然能靠代码检查清楚
- 不把广泛的生命周期耦合引入 runtime

### Rust 原型或对照实现

目标：

- 将 Rust 用作高风险方向的原型或对照工具，而不是默认的重写方向

进入主线前至少应满足：

- 原型回答了一个明确的 runtime 或性能问题
- 构建和调试复杂度仍然隔离在主 Go 代码库之外

## 近期主线暂不纳入的内容

这些方向目前不适合作为默认贡献目标：

- 没有可测量正确性或性能收益支撑的大型抽象化重构
- 本可内部解决却扩大 public API 的改动
- 把实验和生产变更混在一起的大范围 runtime 重写
- 没有明确实验目标的跨语言核心路径重写

## 如何高效贡献

高质量贡献通常具备这些特征：

- 一个 PR 只做一个清晰的行为变化
- 正确性改动带测试
- 性能敏感改动带 benchmark 说明
- 公开行为、配置或示例变化时同步更新文档

如果改动涉及 loop ownership、listener 行为、TLS / kTLS 数据路径或 API 扩展，建议先开 design discussion，或者先提交一个小型准备性 PR。
